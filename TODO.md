# TODO: Rewrite Authentication Middleware in C#

The goal of this task is to replace the Python `require_token` decorator with a high-performance C# (ASP.NET Core) middleware. This will reduce latency for every authenticated request by optimizing JWT verification and database user validation.

## 🎯 Objectives
- **Zero Downtime:** Must be 100% compatible with existing Python-generated JWTs.
- **Performance:** Utilize C#'s asynchronous capabilities and efficient database connection pooling.
- **Security:** Maintain strict ban checks and role-based access control.

## 🛠 Technical Stack
- **Framework:** ASP.NET Core 8.0+
- **JWT Handling:** `Microsoft.AspNetCore.Authentication.JwtBearer`
- **Database Access:** Dapper (for maximum performance) or Entity Framework Core.
- **Database:** MySQL (using `Pomelo.EntityFrameworkCore.MySql` or `MySqlConnector`).

## 📋 Implementation Steps

### 1. Project Initialization
- [ ] Create a new ASP.NET Core Web API project.
- [ ] Configure `appsettings.json` to include:
    - `DatabaseSettings:ConnectionString`
    - `JwtSettings:SecretKey` (must match the Python `SECRET_KEY`)
    - `JwtSettings:Issuer` (if applicable)
    - `JwtSettings:Audience` (if applicable)

### 2. JWT Authentication Setup
- [ ] Configure JWT Bearer authentication in `Program.cs`.
- [ ] Ensure `TokenValidationParameters` use the same `SymmetricSecurityKey` and `ValidateIssuer = false`, `ValidateAudience = false` (matching current Python config).
- [ ] Use `HS256` as the security algorithm.

### 3. Database Layer (Dapper)
- [ ] Create a `User` model matching the database schema.
- [ ] Implement a `UserRepository` with a method `GetAuthUserByIdAsync(int id)`:
    - Query: `SELECT id, role, is_banned, username FROM users WHERE id = @Id`
    - Use async/await for non-blocking I/O.

### 4. Custom Authentication Middleware / Handler
- [ ] Create a custom `JwtBearerEvents` or a dedicated Middleware to perform the database check AFTER the token is validly decoded.
- [ ] **Logic:**
    1. Extract `id` from JWT claims.
    2. Fetch user from DB.
    3. If user doesn't exist -> `401 Unauthorized`.
    4. If `is_banned` is true -> `403 Forbidden`.
    5. Attach `id`, `username`, and `role` to the `HttpContext.Items` or as additional `Claims` for downstream use.

### 5. Role-Based Authorization
- [ ] Implement `[Authorize(Roles = "admin")]` and `[Authorize(Roles = "mod,admin")]` equivalents.
- [ ] Ensure roles are correctly mapped from the database string to C# Claims.

### 6. Compatibility & Testing
- [ ] Verify that a JWT generated by the Python `create_jwt` function is successfully validated by the C# middleware.
- [ ] Verify that an expired Python JWT is rejected.
- [ ] Verify that a banned user is rejected even with a valid JWT.

### 7. Deployment Strategy
- [ ] Set up a reverse proxy (like Nginx) to route specific paths to the new C# service while keeping others on Python.
- [ ] Alternatively, use the C# service as an "Authentication Gateway" that validates tokens before forwarding requests to the Python backend.

## ⚠️ Key Considerations
- **PIN Hashing:** If any C# parts need to check PINs, remember it uses `bcrypt` on top of a `sha256` material: `bcrypt(sha256(pin:uuid:PIN_SALT))`.
- **Environment Variables:** Ensure `SECRET_KEY` and `PIN_SALT` are securely managed in the C# environment.
